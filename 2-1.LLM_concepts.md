# Phase 2-1. LLM 기초 개념 & API 빌드

## 1-1. LLM의 본질 : "다음 토큰 예측기"

LLM = 기존 토큰 순서(sequence) 주어졌을때, 그 다음 어떤 토큰이 올 확률이 높은지를 출력하는 함수

> LLM이란 이 매커니즘을 반복하는 결과일 뿐이다.

### 어떤 관점에서 봐야 하는지?
    1. Halluciation이 발생하는 이유?
    : LLM은 그 다음에 올 확률이 높은 토큰을 뽑아주는 역할만 수행함. (사실 검증이 누락됨.)
        - RAG가 도입되면서 완화됨.
    2. 왜 같은 질문에 다른 답?
    : 확률 무작위 생성 특성 때문이다. 
    - 조절하는 방법 : temperature(온도) - 1-4. Decoding에서 확인
        사실 확인 - 0.0 ~ 0.3 - 정확하고 일관된 답변
        코드 생성/수정 - 0.2 ~ 0.5 - 구조적인 정확성이 중요
        일반적 대화 - 0.7 ~ 0.8 - 적당히 자연스럽고 유연한 대화
        창작(시/소설) - 1.0 ~ 1.5 - 의외 표현과 예술적 허용


## 1-2. Tokenization (토큰화)

LLM은 토큰 단위로 처리 (모든 비용/제약 단위)
(한국어는 토큰이 비싸다)

영어 1단어 = 1.3토큰
한국어 1단어 = 1 ~ 2 토큰 (약 2 ~ 3배 비쌈)

### 실무에서 token을 모르면 생기는 일?
    1. 우리 LLM 프로그램이 비싼 이유를 모름.
    2. Context window 200K인데 왜 잘림?
    : 한국어는 토큰을 많이 잡아먹는다.
    3. cache hit rate가 낮다. - 미세하게 조정되는 경우 프롬프트 캐싱이 안됨.
    (프롬프트 캐싱 : LLM 호출시 변하지 않는 내용은 메모리에 보관했다가 동일한 내용이 입력되면 보관된 정보를 재사용)
    (hit : 캐시된 영역이 재사용되는 경우)


## 1-3. Context Window
: LLM이 한 번에 볼 수 있는 토큰 최대 수

| 모델 | Context Window |
| --- | --- |
| Claude Sonnet 4.5 / Opus 4.x | 200K |
| GPT-4o | 128K |
| Gemini 1.5 Pro | 1M ~ 2M |

- 비용 계산 공식
    비용 = (input_tokens x input_price) + (output_tokens x output_price)

    : 일반적으로 output이 input보다 5배 비쌈
    
    > TIP : 응답을 짧게 많드는 프롬프트가 비용 절감
    (매 요청마다 전체 대화 히스토리를 다시 보냄 - 대화가 길어질수록 비용이 누적 증폭)

### LLMOps의 핵심
: Prompt Caching, Batch API, 작은 모델로 라우팅 등 최적화!


## 1-4. Decoding 파라미터 - 응답을 통제하는 손잡이
: LLM 출력은 확률분포에서 샐플링됨. (샘플링 방식을 제어하는 파라미터들)

1. Temperature (온도)
> 확률 분포의 "뾰족함"을 조절, 낮을수록 결정적, 높을수록 다양성

| Temperature | 동작 | 사용처 |
|---|---|---|
| 0.0 | 항상 가장 높은 확률 토큰 | SQL 생성, JSON 추출, 분류 |
| 0.3~0.5 | 약간의 다양성 | 일반 QA, 요약 |
| 0.7~1.0 | 다양성 큼 | 창작, 브레인스토밍 |
| 1.0+ | 매우 자유로움 | 거의 안 씀 (혼란스러움) |

> TIP : 실무 가이드
    불확실하면 0.0부터 시작
    응답이 너무 똑같으면 0.3 ~ 0.5로 조정

2. Top-p (Nucleus Sampling, 0.0 ~ 1.0)
> 누적 확률이 p에 도달할 때까지의 토큰만 샐플림 후보로 둠
즉, p값(임계값)에 따라 후보군 수가 달라짐. (누적 확률이 특정 임계값에 도달할 때까지 상위 후보군을 선택하는 기법)

p값이 높을수록 후보군이 넓어져 다양하고 창의적인 답변 제공
p값이 낮을수록 후보군이 엄격해져 논리적 문장 제공
(일반적으로 0.7 ~ 0.9로 설정)

- Temperature와 Top-p는 보통 둘 중 하나만 조절해서 사용 (아니면 꼬임)

3. Top-k
> 상위 k개 토큰만 후보로 (Anthropic은 기본 사용 X, OpenAI도 잘 사용 안함)
즉, Top-p처럼 작동하나, 상위 후보군 수 중 몇개를 포함할지 결정하는 방식임.

4. Max Tokens
> 출력 최대 토큰 수. 비용/지연시간 통제의 가장 직접적인 손잡이

5. Stop Sequences
> 특정 문자열이 나오면 즉시 중단. 구도화된 출력에 유용

```python
stop_sequences=["</answer>", "\n\nUser:"]
```


## 1-5. Messages 구조 - Role 기반 대화
: 현대 LLM API의 표준 입력 형식

```python
messages = [
    {"role": "user", "content": "안녕"},
    {"role": "assistant", "content": "안녕하세요!"},
    {"role": "user", "content": "오늘 뭐할까?"}
]
```

| Role | 역할 |
|---|---|
| `system` | 모델의 정체성/규칙 정의 (Anthropic은 별도 `system` 파라미터) |
| `user` | 사용자 입력 |
| `assistant` | 모델의 응답 (multi-turn 시 이전 답변을 다시 넣어줌) |
| `tool` / `tool_result` | 함수 호출 결과 (Tool Use) |

- LLM은 기억이 없다.
: API는 stateless (서버에 대화 저장하지 않음)
: 매번 전체 대화 히스토리를 user가 직접 보내야 multi-turn 가능
: "메모리 기능"은 전부 클라이언트(우리 쪽) 구현


## 1-6. System Prompt - 모델 정체성 정리

역할
- 페르소나 ("당신은 python 개발자입니다")
- 출력 형식 ("응답은 항상 JSON으로 설정합니다")
- 제약 ("불확실하면 모른다고 답변하라.")
- 도구 사용 규칙

### System Prompt 작성 베스트 프랙티스
1. 첫 줄에 역할 정의 1문장
2. 목표/제약을 bullet으로 
3. 출력 형식 예시
4. 너무 길면 효과 떨어짐 (1500토큰 이상은 약효 감소)
5. XML 태그로 섹션 구분 (Anthropic 모델에 특히 효과적)

- 예시 (XML 태그 설명)

<role>
너는 복잡한 비즈니스 데이터를 분석하여 의사결정에 필요한 핵심 인사이트를 도출하는 10년 차 전략 컨설턴트야.
</role>

<goals_and_constraints>
* **목표:** 데이터의 수치적 변화를 넘어 그 뒤에 숨겨진 '이유'와 '향후 전략'을 제시할 것.
* **제약:** 전문 용어는 사용하되, 고등학생도 이해할 수 있도록 비유를 섞어 설명할 것.
* **금지:** 확실하지 않은 통계적 추측은 지양하고, 데이터에 기반한 사실만 서술할 것.
</goals_and_constraints>

<format_example>
### 📊 분석 요약
* **핵심 지표:** [지표명]: [수치] ([상태])
* **전략 제언:** [한 줄 요약]

### 💡 상세 인사이트
1. [인사이트 제목]
   - 내용 설명...
</format_example>

<instruction>
이제 사용자가 제공하는 매출 데이터를 바탕으로 보고서를 작성해줘.
</instruction>


## 1-7. Streaming - 실시간 응답
- 필요한 이유?
: LLM 응답은 보통 5 ~ 30초 정도 소요됨. (한 번에 받으면 UX 최악)
-> 토큰 단위로 스트리밍하면 ChatGPT처럼 글자가 흘러나옴

"지각된 성능(Perceived Performance)" : 글자가 나오고 있으면 "작동하고 있구나"라는걸 알 수 있음.


## 1-8. Toole Use / Function Calling - LLM에 손과 눈 달기
: LLM에게 함수 명세서를 전달하면 필요할 때 "이 함수를 이런 인자로 호출해줘"라고 응답함. 우리가 실제 함수를 실행하고 결과를 다시 LLM에게 돌려줌.

- 예시 flow
    1. user : "내일 서울 날씨 어때?"
    2. LLM 응답 (tool_use) : get_weather(city="Seoul", date="2026-04-26")
    3. 우리가 실제 함수 실행 -> "맑음 18도"
    4. LLM에 tool_result로 다시 전달
    5. LLM에 응답(text) : "내일 서울은 맑고 18도 예상됩니다."

### 왜 게임 체인저인가?
    1. DB쿼리, API호출, 계산 등 LLM이 못하는 일을 위임 가능
        : 주식, 날씨 등 최근 정보들을 직접 호출할 수 있음.
    2. Agent 시스템 기반
        : user가 요청하는 목표를 해결하기 위해 AI가 스스로 계획을 세우고 실행하도록 제공
    3. Structered Output 응용
        : 명확한 규격을 제공하면 해당 규격에 맞춰서 제공해줌.


## 1-9. Structured Output - 신뢰 가능한 JSON 받기
: "JSON으로 답해줘"라고 하면 모델이 가끔 마크다운 코드블록을 씌우거나 누락 필드 발생함.

- 해결책
1. 프롬프트 + 파싱 (가장 약함)
prompt = "다음 형식의 JSON으로만 답하세요: {...}"

2. Tool Use 활용 (Anthropic 권장)
: 함수 정의에 schema 넣고 강제로 그 형식으로 응답하게 함.
: 단일 tool 정의 + 강제 사용

3. Structured Output 모드 (OpenAI response_format + Pydantic)

4. Instructor 라이브러리 : 프로바이더 통합 (강력 추천)
> 재시도 로직 : 파싱 실패 시 자동으로 모델에게 오류 알려주고 재실행
> 일관성 있음.
> pydantic 통합

## 1-11. 모델 선택 가이드 (2026년 4월 기준)

#### Anthropic 라인업 (대략적 포지션)
| 모델 | 강점 | 사용처 |
|---|---|---|
| **Claude Opus 4.x** | 최고 성능, 복잡한 추론 | 어려운 분석, 코드 생성, 에이전트 핵심 |
| **Claude Sonnet 4.x** | 균형 (속도·품질·비용) | **대부분의 실무 디폴트** |
| **Claude Haiku 4.x** | 빠르고 저렴 | 분류, 라우팅, 간단 응답 |
 
#### OpenAI 라인업
| 모델 | 강점 | 사용처 |
|---|---|---|
| GPT-4o / o1 / o3 | 추론 / 멀티모달 | 복잡한 문제 |
| GPT-4o-mini / GPT-5-nano급 | 저렴 | 분류, 단순 응답 |
 
#### 모델 선택 의사결정 트리
```
1. 분류/라우팅/간단 응답?     → Haiku / mini
2. RAG 응답 생성?              → Sonnet
3. 복잡한 추론 (코드/분석)?    → Opus / o-시리즈
4. 비용이 가장 중요?           → 가장 작은 모델부터 테스트 후 올리기
```
