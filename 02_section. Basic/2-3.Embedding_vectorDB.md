# Phase 2-3. 임베딩 & 벡터 DB 기초

## 왜 임베딩이 RAG의 70%인가?
RAG 시스템이 실패하는 이유의 약 70%는 LLM의 답변 능력이 아니라,
"애초에 적절한 문서를 못 가져왔기 때문"이다.
"어떤 문서를 가져올까?"를 결정하는 것이 임베딩 + 벡터 DB
이 단계가 부실하면 LLM이 아무리 똑똑해도 잘못된 정보로 답변

- Growth x AI Builder에게:
1. 사내 문서 챗봇 -> 임베딩
2. 유저 피드백 클러스터링 -> 임베딩
3. 유사 케이스 추천 -> 임베딩
4. A/B 실험 결과 검색 -> 임베딩
> 모든 LLM 응용의 입구가 임베딩


## 1. 이론 및 개념
### 1-1. 임베딩(Embedding)의 본질
: 임베딩 = 텍스트(또는 이미지/오디오)를 고정 차원의 실수 벡터로 변환한 것.
> 의미가 비슷한 것들은 벡터 공간에서 가까이 위치함.

예시)
"강아지" → [0.21, -0.45, 0.88, ..., 0.13]   # 1024차원 (예시)
"개"     → [0.20, -0.43, 0.87, ..., 0.12]   # "강아지" 와 매우 가까움
"자동차" → [-0.55, 0.71, 0.02, ..., -0.34]  # 멀리 떨어짐

> 왜 이게 마법인가?
1. 텍스트의 의미를 좌표로 표현 -> 수학(거리, 유사도)으로 의미 비교 가능
2. 단순 키워드 매칭의 한계를 넘음 ("강아지 키우기" 검색 -> "반려견 양육법"도 매칭)
3. 다국어 임베딩이면 한국어로 영어 문서 검색도 가능


### 1-2. 임베딩의 5가지 핵심 속성
#### 1. 차원 (Dimension)
> OpenAI text-embedding-3-small: 1536
> OpenAI text-embedding-3-large: 3072 (또는 축소 가능)
> BGE-m3, multilingual-e5: 1024
> 한국어 KoSimCSE: 768

#### 2. 정규화 (Normalization)
: 대부분의 임베딩의 L2-norm = 1 (단위 벡터)
> 코사인 유사도 == 내적 (계산 빠름)
> 정규화 안 된 임베딩은 거리 비교 시 주의

#### 3. 도메인 적합성
> 일반 임베딩 (multilingual-e5): 다용도
> 도메인 특화 (법률, 의료): 해당 도메인 점수 더 높음
> 코드 임베딩 (CodeBERT, Voyage code-3): 코드 검색 허용

#### 4. 다국어 지원
> 한국어 단독: KoSimCSE, ko-sroberta
> 다국어(한국어 포함): multilingual-e5, BGE-m3, OpenAI 3-large 등

#### 5. 비용
> API 호출 단위: 1M 토큰당 $0.02~$0.13 (모델별)
> 로컬 모델: GPU비용 + 운영 부담, 하지만 호출 무제한


### 1-3. 거리와 유사도
- 세 가지 주요 측정법:

1. 코사인 유사도(Cosine Similarity) - 가장 흔함
sim(A, B) = (A · B) / (|A| × |B|)
> 범위: -1 ~ 1 (1=동일, 0=무관, -1=정반대)
> 길이 무시 -> 길이가 의미 없는 임베딩에 적합 (대부분 케이스)

2. 유클리드 거리(L2)
dist(A, B) = √Σ(Aᵢ - Bᵢ)²
> 범위: 0 ~ ∞ (0=동일)
> 정규화된 임베딩에서는 코사인과 단조 관계

3. 내적(Dot Product)
A · B = Σ AᵢBᵢ
> 정규화된 벡터에서는 코사인 유사도와 동일
> 가장 빠른 계산

- 실무권장 : 거의 항상 코사인 유사도 or 정규화 후 내적 사용
> Vector DB들도 디폴트로 cosine

### 1-5. 벡터 DB (Vector Database)의 역할
- 왜 일반 DB로는 안 되나?
1. 1만 개 문서  → 매번 모든 벡터와 코사인 계산 = O(N) → 느림
2. 벡터 DB는 ANN (Approximate Nearest Neighbor) 인덱스로 O(log N) 근방
3. HNSW, IVF, ScaNN 같은 알고리즘 내장

- 주요 벡터 DB 비교
| DB | 타입 | 특징 | 추천 시점 |
|---|---|---|---|
| **Chroma** | 로컬/임베디드 | 가장 간단, SQLite 기반 | **PoC, 학습** |
| **pgvector** | PostgreSQL 확장 | SQL 쿼리 + 벡터 검색 결합 | 이미 PostgreSQL 쓰는 팀 |
| **Qdrant** | 독립 서버 | 빠르고 기능 풍부, 오픈소스 | 프로덕션 디폴트 |
| **Weaviate** | 독립 서버 | 스키마 기반, GraphQL | 메타데이터 많을 때 |
| **Milvus** | 독립 서버 | 대규모, 분산 | 1억 벡터 이상 |
| **Pinecone** | 매니지드 SaaS | 운영 부담 0 | 빠르게 출시 |
| **Elasticsearch / OpenSearch** | 검색엔진 | BM25 + Vector 통합 | Hybrid Search 핵심 |

- 학습 권장 순서
Chroma (개념 이해, 1주)
  ↓
pgvector (SQL 결합 학습, 1주)
  ↓
Qdrant (프로덕션 패턴, 본인 프로젝트에 도입)


### 1-6. ANN (Approximate Nearest Neighbor) 알고리즘
> 핵심 아이디어 : 정확한 1등 대신 거의 1등을 빠르게 찾자.

1. HNSW (Hierarchical Navigable Small Word) - 가장 흔함
> 그래프 기반, 다층 구조
> 빠르고 정확도 높음
> Chroma, Qdrant, pgvector 디폴트

2. IVF (Inverted File Index)
> 클러스터링 기반
> 메모리 효율 좋음, 거대 데이터에 적합

3. Flat (Brute Force)
> 모든 벡터와 비교 (정확하지만 느림)
> 작은 데이터(<1만) 또는 정확도 100% 필요시

4. 튜닝 파라미터
> HNSW M, ef_construction, ef_search - 정확도 vs 속도
> 처음에는 디폴트로 시작, 응답 품질 부족 시 ef_search부터 올림


### 1-7. 메타데이터와 필터링
> 벡터만으로는 부족 → 메타데이터로 보완.
```python
{
    "text": "Phase 2 학습 자료...",
    "embedding": [0.1, 0.2, ...],
    "metadata": {
        "source": "Phase2-1.md",
        "category": "LLM",
        "created_at": "2026-04-25",
        "language": "ko",
        "page": 12
    }
}
```

- 필터링 검색 예시
"language=ko AND created_at>=2026-01-01"인 것 중에서 "임베딩 차원"과 의미적으로 가까운 top5
→ Vector DB 들은 거의 모두 메타데이터 필터링 지원.


### 1-8. 임베딩 캐싱
> 문제: 동일 텍스트를 매번 임베딩하면 비용 낭비.

- 해결
```python
import hashlib, json
from pathlib import Path
 
CACHE_DIR = Path("./embedding_cache")
CACHE_DIR.mkdir(exist_ok=True)
 
def get_embedding_cached(text: str, model: str) -> list[float]:
    key = hashlib.md5(f"{model}::{text}".encode()).hexdigest()
    cache_file = CACHE_DIR / f"{key}.json"
    
    if cache_file.exists():
        return json.loads(cache_file.read_text())
    
    emb = compute_embedding(text, model)  # 실제 API 호출
    cache_file.write_text(json.dumps(emb))
    return emb
```
> 프로덕션에서는 Redis 또는 DB 활용.


### 1-9. 임베딩의 한계
⚠️ 임베딩이 못 하는 것들:
1. 정확한 키워드 매칭("문서 ID 12345 찾아줘") → BM25 가 더 잘함
2. 부정 표현("X가 아닌 것") → 벡터 공간에서 잘 표현 안 됨
3. 숫자 비교("100만 원 이상") → 메타데이터 필터로 해결
4. 시간 추론 ("최근 1주일") → 메타데이터 필터로 해결
5. 다단계 reasoning → LLM 의 영역

→ 그래서 Hybrid Search (Vector + BM25) 가 표준이 됨 (Phase 2-4).


### 1-10. 한국어 임베딩의 특수성
- 토크나이저 차이
> 영어 모델: 한국어를 잘게 쪼개서 처리 → 의미 손실 가능
> 한국어 학습 모델 (BGE-m3, KURE): 형태소 단위 이해

- 띄어쓰기·맞춤법 영향
> 좋은 임베딩 모델은 어느 정도 흡수
> 하지만 정확도 위해 사전 정규화 권장(`hangul-utils`, `soynlp` 활용)

- 평가 지표
> KO-MTEB(한국어 MTEB) - 한국어 임베딩 벤치마크
> 본인 도메인에 맞는 평가셋으로 직접 측정 권장
