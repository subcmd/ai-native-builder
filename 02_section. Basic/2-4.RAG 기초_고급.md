# Phase 2-4. RAG 기초 -> 고급

## 목표
1. RAG의 전체 파이프라인을 직접 구현할 수 있다.
(Retrieval → Augmentation → Generation)
2. 청킹·검색·재정렬·쿼리재작성을 단계적으로 개선하는 사이클을 돈다
3. Naive RAG의 한계와 Advanced RAG 패턴 (Hybrid, Rerank, HyDE 등)을 안다
4. RAG 시스템을 숫자로 평가하고 개선할 수 있다.

## RAG가 LLM 응용의 80%인 이유?
- LLM 단독의 한계:
> 학습 시점 이후 정보 모름 (knowledge cutoff)
> 사내/개인 데이터 모름
> 환각 (hallucination)
> 출처 추적 불가

- RAG = "외부 지식을 검색해 LLM 컨텍스트에 주입"으로 위 모두 해결.
"Anthropic·OpenAI 도 인정: 2025년 기준 프로덕션 LLM 응용의 70~80%가 RAG 기반."

- Growth x AI Builder에게 RAG는:
> 사내 PRD/문서 챗봇: 파편화된 제품 지식의 중앙 집권화
(제품 요구사항 정의서 PRD: 제품 목적, 기능, 사용자 경험 등 핵심 정의 담은 문서)
> 고객 CS 자동 응답: 단순 반복 문의의 완전 자동화와 정확도 확보
> A/B 실험 결과 검색: 성공과 실패의 기록을 자산으로 전환
> 데이터 카탈로그 자연어 질의: 데이터 민주화 - SQL을 몰라도 가능한 데이터 탐색


## 1. 이론 및 개념
### 1-1. RAG의 본질 - 3단계 파이프라인
```
[User Query]
     ↓
1) Retrieval     → 관련 문서를 외부 DB에서 검색
     ↓
2) Augmentation  → 검색된 문서를 프롬프트에 주입
     ↓
3) Generation    → LLM이 컨텍스트 기반으로 답변
     ↓
[Answer + Sources]
```

- 가장 단순한 형태:
```python
def naive_rag(query: str) -> str:
  docs = vector_db.search(query, k=3)
  prompt = f"""다음 문서를 참고해 질문에 답하시오.

<documents>
{docs}
</documents>

<question>
{query}
</question>"""
    return llm.complete(prompt)
```
> 이게 RAG의 핵심. 나머지는 전부 각 단계의 개선

---

### 1-2. RAG 진화 단계

| 단계 | 이름 | 특징 | 예시 |
|---|---|---|---|
| 1세대 | **Naive RAG** | 단순 청킹 + 벡터 검색 + 단순 프롬프트 | 도서관에서 키워드로 책 찾아 읽어주기 |
| 2세대 | **Advanced RAG** | Pre/Post-retrieval 최적화, Hybrid Search, Re-rank | 사서가 관련성 높은 책을 골라 순서대로 정렬해주기 |
| 3세대 | **Modular RAG** | 라우팅, 다중 검색, Agent 결합 | 질문에 따라 전문 도서관 혹은 일반 도서관 선택하기 |
| 4세대 | **Agentic RAG** | LLM 이 검색 전략까지 결정 | 비서가 직접 조사하고 분석해서 결과 보고서 가져오기 |
 
- 현실적인 학습 순서
> Naive → Advanced까지가 본 Phase.
> Modular/Agentic 은 Phase 2-5(Agent)에서.

### 1-3. 청킹(Chunking) 전략
왜 쪼개나? 검색 정확도 + 컨텍스트 윈도우 효율 + 비용
(컨텍스트 윈도우: AI가 한 번에 뇌에 담아두고 처리할 수 있는 정보의 시야) = 범위를 확인하는 스캐너

1. Fixed-size Chunking
> 글자 수 또는 토큰 수 단위
> 장점: 단순
> 단점: 문장/문단 중간에 끊김

2. Recursive Character Splitting (LangChain 디폴트)
> 문단 → 문장 → 단어 순으로 자연스러운 경계 우선 분할
> 실무 권장 디폴트

```python
from langchain_test_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
  chunk_size=800,     # 약 400~600 한국어 글자
  chunk_overlap=100,  # 청크 간 중복 (맥락 유지)
  separators=["\n\n", "\n", ". ", "? ", "! ", " ", ""]
)
chunks = splitter.split_text(long_text)
```

3. Sentence Splitting
> 문장 단위로 분할 → 작은 청크
> 짧은 답변 검색에 유리

4. Semantic Chunking
> 임베딩 유사도가 떨어지는 지점에서 분할
> 자연스러운 의미 경계
> 비용 ↑ (각 문장 임베딩 필요)

5. Document-aware Chunking
> 마크다운 헤더, HTML 태그, 코드 블록 단위로 분할
> 추천: 마크다운 문서 RAG에 효과적

```python
from langchain_text_splitters import MarkdownHeaderTextSplitter

splitter = MarkdownHeaderTextSplitter(
  headers_to_split_on=[("#", "h1"), ("##", "h2"), ("###", "h3")]
)
```

6. Parent-Child / Hierarchical Chunking ⭐ 고급
> 작은 청크(child)로 검색 → 큰 청크(parent)를 LLM 에 전달
> "정확한 매칭은 작게, 컨텍스트는 크게"의 절충
> LlamaIndex `SentenceWindowNodeParser`, `AutoMergingRetriever`

---

### 1-4. 청크 사이즈 가이드

| 사용처 | 권장 chunk_size (토큰) | 권장 overlap |
|---|---|---|
| QA (정확한 사실 검색) | 256~512 | 50~100 |
| 문서 요약 | 512~1024 | 100~200 |
| 코드 검색 | 1024~2048 (함수 단위) | - |
| 한국어 문서 | 영어보다 1.5~2배 작게 | - |
 
🚨 한국어 주의
: 토큰 수 ≠ 글자 수. 한국어는 토큰 인플레이션 큼.
> chunk_size=800 토큰 ≈ 한국어 400~500자

### 1-5. Retrieval 단계 - 4가지 검색 전략

1. Vector Search (Dense Retrieval)
> 임베딩 기반 의미 검색
> 강점: 의미적 유사성, 동의어
> 단점: 정확한 키워드, 고유명사

2. Keyword Search (Sparse Retrieval, BM25)
> TF-IDF기반 + 길이 정규화
> 강점: 정확한 매칭, 희귀 단어
> 단점: 동의어, 의역

3. Hybrid Search ⭐ 표준
> Vector + BM25 결과를 결합
> 결합 방식: 가중치 합산 또는 RRF (REciprocal Rank Fusion)

```python
def reciprocal_rank_fusion(results_lists, k=60):
    """여러 검색 결과 리스트를 RRF로 결합"""
    scores = {}
    for results in results_lists:
        for rank, doc_id in enumerate(results):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: -x[1])
```

- Q: 왜 Hybrid이 거의 항상 더 나은가?
> Vector가 놓치는 케이스(제품명, 코드, 숫자)를 BM25가 잡음
> BM25가 놓치는 케이스(동의어, 의역)를 Vector가 잡음
> 두 신호의 보완성

4. Re-ranking
> 1차 검색 결과(top50)를 Re-ranker 모델이 재정렬 → top 5
> Cross-encoder 기반 → 정확하지만 느림 → 1차 결과만 대상으로
> Cohere Rerank, BGE Reranker, Jina Reranker, Voyage Rerank

### 1-6. Pre-retrieval 최적화 (Query 측 개선)

1. Query Rewriting
> 사용자 질문을 검색에 적합하게 변환:
```
원본: "iPhone 15 Pro 의 카메라 광학 줌 배율은?"
Step-back: "iPhone 15 Pro 의 카메라 사양"
```

2. Multi-query
> 한 질문을 여러 변형으로 검색 → 결과 합치기:
```
원본: "리텐션 어떻게 측정해?"
변형1: "Day-N retention 정의"
변형2: "리텐션 코호트 분석 방법"
변형3: "user retention metrics"
```

3. HyDE (Hypothetical Document Embeddings)
> 아이디어: LLM이 "가상의 답변"을 먼저 생성 → 그 답변으로 임베딩 검색
> 질문보다 답변 형태가 문서와 의미적으로 더 가까움
> 비용 ↑, 효과 ↑↑ (특히 짧은 질문)
```python
def hyde_search(query: str):
    # 1) LLM이 가상의 답변 생성
    hypothetical = llm.complete(f"이 질문에 대한 답변을 작성: {query}")
    # 2) 답변으로 임베딩 검색
    return vector_db.search(hypothetical, k=5)
```

4. Step-back Prompting
> 구체적 질문 → 더 일반적인 질문으로 한 단계 위로 → 검색
```
원본: "iPhone 15 Pro의 카메라 광학 줌 배율은?"
Step-back: "iPhone 15 Pro의 카메라 사양"
```

---

### 1-7. Post-retrieval 최적화

1. Re-ranking (1-5에서 다룸)
> 1차 top50 → Reranker → top 5

2. Compression / Summarization
> 검색된 청크를 LLM으로 압축 → 컨텍스트 절약

3. Diversification (MMR)
> Maximum Marginal Relevance: 유사도 + 다양성 조합
> 비슷한 청크가 top5를 다 차지하는 것 방지

4. Context Reordering
> LLM은 컨텍스트의 처음과 끝에 있는 정보를 가장 잘 활용 ("Lost in the middle")
> 가장 중요한 청크를 처음·끝에 배치

### 1-8. Generation 단계 - 프롬프트 패턴

```xml
<system>
당신은 사내 문서 QA 어시스턴트입니다.
다음 규칙을 엄수하시오:
1. 제공된 컨텍스트만 근거로 답변
2. 컨텍스트에 없으면 "문서에 해당 정보가 없습니다"라고 답변
3. 답변 끝에 [Source:  파일명] 형식으로 출처 표기
4. 추측·일반 지식은 사용하지 않음
</system>

<user>
<context>
{retrieved_chunks_with_sources}
</context>

<question>
{user_query}
</question>

<o>
</user>
```

#### 핵심 원칙
- 출처 강제 인용 → 검증 가능성
- "모른다" 허용 → 환각 방지
- 컨텍스트 외부 지식 금지 명시

### 1-9. RAG 평가 (Phase 2-7 기초)

#### 평가 차원 4가지
1. Retrieval Quality: 적절한 문서를 가져왔는가?
2. Faithfulness: 답변이 컨텍스트에 충실한가? (환각 없음)
3. Answer Relevance: 답변이 질문에 적절한가?
4. Context Relevance: 가져온 컨텐스트가 질문과 관련 있는가?

#### 측정 방법
- Recall@k: 정답 문서가 top k에 들어왔는가?
- MRR (Mean Reciprocal Rank): 정답 문서의 평균 역순위
- NDCG: 순위까지 고려한 점수
- RAGAS: (오픈소스 라이브러리)
- LLM-as-Judge: 답변 품질 자동 채점 (Phase 2-7)

#### Golden Set 구성 (필수)
```jsonl
{"query": "DAU 정의", "expected_docs": ["doc12"], "expected_answer_keywords": ["일간", "활성"]}
{"query": "리텐션 측정법", "expected_docs": ["doc45", "doc67"], "expected_answer_keywords": ["코호트", "Day-N"]}
```
**🚨 Golden Set 없이 RAG 개선은 불가능**. 반드시 30~100개 작성.

### 1-10. RAG 디버깅 체크리스트

응답이 이상할 때 순서대로:

1. **검색 결과 출력** → 적절한 청크가 들어왔나?
2. **청킹 시각화** → 청크가 의미 있는 단위인가?
3. **임베딩 모델 변경** (BGE-m3 ↔ OpenAI)
4. **Hybrid Search 도입** → BM25 추가
5. **Re-ranker 추가**
6. **chunk_size 조정** (작게 / 크게)
7. **Multi-query / HyDE 도입**
8. **System prompt 강화** (출처 인용, 모른다 허용)
9. **Generation 모델 업그레이드** (Sonnet → Opus)

**디버깅 1순위**: "검색이 안 된 거냐, 검색은 됐는데 LLM 이 못 쓴 거냐" 를 분리.
- 검색 결과 점검 → 정답 청크 있는데 답변 잘못됨? → Generation 문제
- 정답 청크 없음? → Retrieval 문제

---

### 1-11. RAG 운영 시 자주 만나는 함정 12가지

1. 청킹 너무 작음 → 컨텍스트 부족
2. 청킹 너무 큼 → 매칭 정확도 ↓
3. Overlap 없음 → 청크 경계의 정보 손실
4. 메타데이터 없음 → 필터링 불가, 출처 추적 불가
5. 임베딩 모델 부적합 (영어 모델로 한국어)
6. 단일 쿼리만 검색 (Multi-query 미사용)
7. Re-ranker 없음 (top-k가 noise 많음)
8. 출처 인용 강제 안 함 → 환각 발생
9. "모른다" 허용 안 함 → 억지 답변
10. 컨텍스트 너무 많이 → "Lost in the middle"
11. 인덱스 갱신 정책 없음 → stale data
12. 평가셋 없음 → 개선이 측정 불가

---

### 1-12. 언제 RAG를 쓰지 말아야 하는가?

❌ **RAG 부적합**:
- 답변이 문서에 직접 안 나오고 계산/추론이 필요 → Tool Use (Phase 2-5)
- 데이터가 작고(수십 개) 거의 안 변함 → 그냥 system prompt 에 다 넣기
- 사용자 질문이 매우 구조화 (form 필드) → SQL/일반 검색
- 정확한 매칭 필수 (제품 ID, 사용자 ID) → DB 직접 조회

✅ **RAG 적합**:
- 문서가 많음 (수백~수십만)
- 자주 갱신됨
- 자연어 질문
- 출처 인용 필요
