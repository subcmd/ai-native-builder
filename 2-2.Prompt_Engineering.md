## 1. 이론 및 개념
### 1-1. 좋은 프롬프트 6가지 구성요소
Anthropic, OpenAI 공식 가이드와 실전 경험에서 공통된 패턴:
1. Role / Identity      → "당신은 ___ 입니다"
2. Task / Goal          → 무엇을 해야 하는가
3. Context              → 배경 정보, 도메인 지식
4. Examples (Few-shot)  → 입력→출력 사례
5. Format / Constraint  → 출력 형식, 길이, 금기
6. Reasoning Trigger    → "단계별로 생각하라" 등

> 모든 프롬프트가 6개 다 필요한 건 아니다. 
> 단순 작업시 1 ~ 3개 정도여도 충분함.
> 응답이 만족스럽지 않은 경우에 6가지를 체크해서 사용해야 함.


### 1-2. XML 태그 구조화 (Anthropic 모델 특히 효과적)
Claude는 학습 과정에서 XML 태그로 구조화된 입력에 강하게 반응하도록 훈련됨.
구분이 명확하고 파싱이 쉬워짐.

- BAD 예시 (구조 없음)
"다음 리뷰가 긍정인지 부정인지 판단하고 이유와 함께 답해줘.
배송이 느렸어요. 그래도 제품은 좋네요
JSON으로 답변해줘."

- GOOD 예시 (XML 구조화)
<task>
다음 고객 리뷰의 감성을 분류하시오.
</task>
<review>
배송이 느렸어요. 그래도 제품은 좋네요.
</review>
<output_format>
<sentiment>positive | negative | neutral | mixed</sentiment>
<reason>판단 근거를 1문장으로</reason>
</output_format>

- 왜 더 좋은가?
1. 모델이 어디까지가 직업 정의/입력/출력 형식인지 명확히 분리
2. Prompt Injection 방어 효과 (사용자 입력을 <review>안에 가둠)
3. 파싱이 정규식 또는 lxml로 쉬움

#### API별 권장 사항
Anthropic: XML 우선, OpenAI에서도 잘 작동함. 
OpenAI는 마크다운 헤더(#, ##)에도 잘 반응함.
-> XML은 두 프로바이더 모두에 잘 작동함.


### 1-3. Zero-shot / Few-shot / Many-shot
> Zero-shot : 예시 없이 지시만
> "다음 문장의 감성을 positive/negative/neutral로 분류"

> Few-shot: 예시 1~5개 제공
<examples>
<example>
<input>배송이 빨랐어요</input>
<output>positive</output>
</example>
<example>
<input>가격이 너무 비싸네요</input>
<output>negative</output>
</example>
</examples>
<input>{user_input}</input>
<output>?</output>

>Many-shot : 수십-수백 개 예시(long-context 모델에서 유효)

#### **선택 가이드**
| 상황 | 권장 |
|---|---|
| 쉬운 분류 (긍/부정) | Zero-shot 도 충분 |
| 도메인 특화 패턴 (PM 보고서 톤 등) | Few-shot 3~5개 |
| 매우 일관된 출력 형식 필요 | Few-shot 5~10개 |
| 복잡한 reasoning | Few-shot + CoT |
| 모호한 작업 | Many-shot (>20) |

#### Few-shot의 함정
1. 예시 갯수가 너무 많으면 그 예시들에만 맞춰서 일반화가 떨어짐
2. 예시들이 한쪽으로 편향되면 응답도 편향됨. (긍정 예시 5개만 -> 부정 입력도 긍정으로 답할 위험)
3. 다양성 + 균형이 핵심


### 1-4. Chain-of-Thought (CoT)
- 원리 : 답을 내기 전에 "단계별로 생각"하게 하면 정확도가 올라감.

> Zero-shot CoT (가장 간단, 강력)
"... 답변하기 전에 단계별로 생각하시오."

또는 영어로 "Let's think step by step"

> Structured CoT (XML 활용)
<instructions>
1. <thinking> 태그 안에서 단계별 분석
2. <answer> 태그 안에 최종 답변
</instructions>

- 왜 효과적인가?
1. LLM은 다음 토큰 예측기 -> 더 많은 "생각" 토큰을 생성하면 다음 답변의 품질이 향상
2. 복잡한 추론(수학, 다단계 SQL, 인과 분석)에서 효과 큼

- 언제 안 쓰는게 나은가?
1. 단순 분류, 추출 -> 오히려 응답이 길어지고 비용 증가
2. 응답 일관성이 중요한 라우팅 -> 추론을 너무 자유롭게 두면 흔들림
3. 최신 reasoning 모델(o-series)은 암묵적으로 CoT 수행 -> 명시적 CoT 지시 불필요할 수 있음.

- Extended Thinking (Claude의 명시적 추론 모드)
1. API 파라미터 thinking={"type": "enabled"}로 활성화
2. 모델이 응답 전 별도 thinking 블록을 생성
3. 복잡한 reasoning 작업에 강력


### 1-5. ReAct 패턴 (Phase 2-5 연결)
- ReAct = Reasoning + Acting

LLM이 다음 사이클을 반복:
"""
Thought: "이 질문에 답하려면 DB 조회가 필요"
Action: query_db("SELECT ...")
Observation: <결과>
Thought: "결과를 바탕으로 분석..."
Action: ...
"""

> Tool Use와 결합하면 자율 에이전트의 기본 패턴이 됨.


### 1-6. 출력 형식 통제
- 옵션1: Pydantic + Structured Output
       : 가장 강함. 타입 안정성 + 검증.
- 옵션2: XML 태그
       : 파싱은 lxml/regex로:
       import re
       match = re.search(r"<sentiment>(.*?)</sentiment>", response)
- 옵션3: JSON 강제
       "다음 형식의 JSON만 출력하세요. 추가 텍스트 금지:
       {\"sentiment\": \"...\", \"score\": ...}"
> 가끔 json 마크다운 블록을 씌움 -> 후처리 필요
> Anthropic에서는 응답을 {로 시작하게 prefill하면 안정적

- Anthropic prefill 패턴
messages = [
    {"role": "user", "content": prompt},
    {"role": "assistant", "content": "{"}  # 응답을 { 로 강제 시작
]


### 1-7. 한국어 프롬프트 특화 패턴
1. 영어 vs. 한국어 시스템 프롬프트
> 영어 system prompt + 한국어 user input + 한국어 응답이 가성비 좋은 경우 많음.
> 이유 : 영어 토큰이 싸고, 모델의 영어 이해도가 더 정밀함.
> 한국어 응답 강제 : Always respond in Korean.

2. 한국어 특유의 모호성 대응
> 존댓말/반말 통제 : "반드시 존댓말로 답하시오."
> 외래어 표기 통제 : "기술 용어는 영문 그대로 두시오"
> 단위 통일 : "원/달러/시간/날짜는 항상 한국식 포맥으로"

3. 한국어 few-shot의 가치
> 한국어는 토큰이 비싸지만, Few-shot 예시 1개의 정확도 기여는 영어보다 큰 경우가 많음.
> 비용 대비 효과가 좋음.


### 1-8. Prompt Injection 방어
